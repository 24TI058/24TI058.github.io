# ソフトウェア工学2026まとめ

　ソフトウェアは「実行されることによって必要な特性、機能、性能を提供する命令語群」であり、ハードウェアと異なり劣化はしないが悪化する。現代社会ではソフトウェアなしでは生活が成り立たず、規模の拡大・複雑化により効率的な開発と品質確保が求められる。また、IT人材不足や社会インフラとしてのソフトウェアの重要性、不確実性の増大などから、体系的なソフトウェア工学の知識が不可欠となっている。
　Googleのソフトウェアエンジニアリングでは、ソフトウェアの持続可能性を強調し、「変更すべき全てのものを安全に変更可能で、かつコードベースの存続期間を通してそれが可能であるとき、組織のコードベースは持続可能である」と述べている。また、Hyrumの法則として「作られたシステムが持つあらゆる観察可能な挙動に依存するユーザーが出てくる」とされ、予期しない使われ方を前提に設計する必要がある。

　ソフトウェアは「誕生 → 開発・運用 → 廃止」というライフサイクルを持つ。
● 誕生
　ニーズの発生から企画・計画を行い、要件定義書を作成する。
要件定義書には目的、機能、性能、構成、制限事項、スケジュールなどが含まれ、プロジェクト関係者の「バイブル」となる。
● 開発・運用
　設計書をもとにコーディングし、テストを行い、リリースする。
稼働後は保守運用を行い、バスタブ曲線（初期故障→安定稼働→磨耗故障）を意識しながら品質を維持する。
● 廃止
　サービス終了や新システムへの移行を行う。

　要件定義はプロジェクトの成否を左右する最重要工程であり、実現する機能・しない機能を明確化する。Feasibility Studyにより実現性を評価し、機能の絞り込みや代替案検討を行う。
　設計では、要件定義をコーディング可能なレベルまで落とし込み、クラスやモジュールの構造を定義する。さらに、WBS（Work Breakdown Structure）を用いて作業を分解し、スコープを明確化することで、プロジェクト管理を容易にする。

　コーディングでは、PythonのPEP8に代表されるスタイルガイドに従い、読みやすいコードを書くことが求められる。資料では「コードは書くよりも読まれることの方が多い」と述べられており、他者や未来の自分が理解しやすいコードを書くことが重要である。
　PEP8では、行の長さ、インデント、スペースの使い方、命名規則などが定義されており、flake8などのLinterを用いて自動チェックすることで品質を高められる。
　品質管理では、バグ発生率やファンクションポイント法などを用いてソフトウェアの規模や品質を評価する。「バグのないソフトウェアはない（と考えるべき）」とされ、計画的なテストと継続的な改善が必要である。

　ソフトウェア開発には複数のプロセスモデルが存在し、状況に応じて使い分ける。
● ウォーターフォールモデル
　要件定義からテストまで順番に進む。進捗管理しやすいが後工程に負荷が集中する。
● スパイラルモデル
　プロトタイプを作成し、フィードバックを得ながら進める。柔軟だが工数が増える。
● 反復型
　機能ごとに分割して開発する。顧客要求を取り入れやすい。
● アジャイル
　変化への対応を重視し、スクラムやカンバン、XPなどの手法を用いる。
　「動くソフトウェアが主たる進捗の確認手段である」とされ、短いサイクルで継続的に価値を提供する。

　プロジェクトは「有期性」と「独自性」を持つ。
　計画の立て方として、フォアキャスティング（積み上げ型）とバックキャスティング（目標から逆算）があり、ソフトウェア開発ではバックキャスティングが有効である。
　Hookerの7原則では、価値提供、シンプルさ、ビジョン、他者理解、未来への開放性、再利用、思考の重要性が示されている。

　gitは分散型バージョン管理システムであり、ローカルでも履歴管理が可能。
コミットにより「いつ・誰が・何を」記録し、ブランチを用いて並行開発を行う。
　GitHubでは、リモートリポジトリ管理、issuesによる課題管理、projectsによるKanban、pull requestによるレビューなどが可能であり、チーム開発の基盤となる。

　GitHub Actionsを用いることで、コードのビルド・テスト・デプロイを自動化できる。
　GitHub Pagesでは、Jekyllを用いてMarkdownからWebページを自動生成でき、開発者はMarkdownを書くのみでよい。

　複数人での開発では非同期コミュニケーションが重要であり、文書化が不可欠である。
　「文書は日本語とは限らない」とされ、英語でのドキュメント作成が求められる場面も多い。

　本講義を通じて、ソフトウェア工学は単なるプログラミング技術ではなく、要件定義、設計、品質管理、プロジェクト管理、バージョン管理、CI/CDなど、ソフトウェアを持続的に開発・運用するための総合的な学問であることを学んだ。現代のソフトウェア開発では、技術力だけでなく、チームで協力し、変化に対応し、品質を維持し続けるための体系的な知識が不可欠である
